# Multi-Agent User Preference Memory and Long-Term Hit Recovery

## Background

In recent multi-agent runs, we saw two related issues:

1. Multi-agent mode could read from the shared long-term memory store, but it did not have a clean path for writing or reusing explicit user preferences.
2. Dashboard metrics often showed `long_term_hit = 0`, which was suspicious because the memory store already contained historical entries, including `user_preference` records.

At the same time, we did **not** want to change the existing single-agent memory workflow. The goal of this round was therefore to improve the multi-agent path only, while keeping single-agent mode as a stable user-selectable option.

## Problem Statement

The previous multi-agent implementation had three gaps:

### 1. User preference memory was not fully wired in multi-agent mode

- The shared long-term memory store already supported `user_preference`.
- Some historical preference entries already existed in the memory data files.
- However, the default multi-agent write policy did not allow the orchestrator to write `user_preference`.
- As a result, multi-agent mode could not reliably capture newly expressed user preferences during task execution.

### 2. Preference injection needed to be relevant, not unconditional

We considered adding a fixed `user preferences` section to the context generated by `build_context()`, but unconditional injection would pollute downstream context. For example, unrelated personal style preferences should not be injected into a purely technical subtask if they are not relevant to the current request.

The correct design is:

- keep a dedicated user-preference section in the multi-agent memory context;
- but only inject items that remain relevant to the current task after preference-specific filtering.

### 3. Long-term memory hit statistics were misleading

During inspection, we found a mismatch between:

- `dm_agent/data/memory/memory_index.meta.json`
- `dm_agent/data/memory/memory_metadata.json`

The vector-store metadata file contained many historical chunks, but the structured metadata index contained only a very small subset of entries. Since `LongTermMemoryStore.search()` maps vector hits back through the in-memory metadata index, many vector hits could not be reconstructed into `MemoryEntry` objects. In practice, this made valid long-term memory retrievals look like misses, which explains why `long_term_hit` stayed unusually low.

## Implementation Changes

### 1. Add explicit user preference support to multi-agent memory

File:

- `dm_agent/multi_agent/memory.py`

Changes:

- Extended `MultiAgentMemoryConfig` with user-preference-specific retrieval and filtering knobs.
- Allowed the `orchestrator` policy to write `user_preference` in addition to `task_summary`.
- Added a dedicated `MemoryWriteTemplate` for `user_preference`.
- Added explicit preference capture at task start via `start_task()`.

New behavior:

- When the user task text contains explicit preference-style statements, the orchestrator can capture them as structured long-term memory candidates.
- This keeps the responsibility centralized. RAGAgent and CodeAgent do not independently write user preferences.

This preserves the multi-agent role boundary:

- `Orchestrator`: coordinates and writes final preference/task-level memory
- `RAGAgent`: reads memory, does not write long-term memory
- `CodeAgent`: writes engineering-oriented memory only

### 2. Add a dedicated but filtered preference section in `build_context()`

File:

- `dm_agent/multi_agent/memory.py`

Changes:

- Reworked long-term context construction from a single raw string into a bundle:
  - general long-term context
  - relevant user preference context
- Added helper methods for:
  - retrieving preference-only candidates
  - extracting query keywords
  - judging preference relevance
  - formatting preference context separately

Important design choice:

This is **not** unconditional injection.

The new `Relevant user preferences` section is only added if:

- the retrieved memory entry is a `user_preference`;
- it passes score and overlap checks;
- it remains relevant to the current task text.

This gives us a stable context slot for user preference memory without forcing unrelated preferences into every sub-agent prompt.

### 3. Record preference hit telemetry separately

File:

- `dm_agent/multi_agent/memory.py`

Changes:

- Long-term lookup events now include:
  - `preference_hit`
  - `preference_count`
- In addition to generic:
  - `long_term_memory_hit`
  - `long_term_memory_miss`
- The hub now also records:
  - `user_preference_memory_hit`
  - `user_preference_memory_miss`

This gives the dashboard enough signal to distinguish:

- overall long-term retrieval behavior;
- user-preference-specific retrieval behavior.

### 4. Tighten write quality rules for `user_preference`

File:

- `dm_agent/multi_agent/memory.py`

Changes:

- `user_preference` is now filtered more strictly than general engineering memory.
- Only explicit preference-like statements are allowed through the quality filter.

This avoids accidental pollution of long-term memory with arbitrary task text that only happens to mention style or workflow indirectly.

### 5. Recover missing metadata from vector-store chunk metadata

File:

- `dm_agent/memory/long_term_memory.py`

Changes:

- Added a metadata recovery pass during `LongTermMemoryStore` initialization.
- If the structured metadata index is stale or incomplete, the store now reconstructs missing `MemoryEntry` items from the vector-store chunk metadata.
- Recovered items are persisted back into the metadata file.

Why this matters:

- Before this fix, the vector store could contain valid long-term memory chunks while the searchable in-memory metadata index remained incomplete.
- After this fix, historical entries are restored into the structured index, so future search results can resolve correctly.

This is the main reason the earlier `long_term_hit = 0` pattern looked suspicious.

## Dashboard Changes

File:

- `dashboard.py`

The multi-agent memory page now includes user-preference-specific visibility:

- `Preference lookups`
- `Preference hit rate`
- per-task / per-agent preference hit breakdown
- richer event table output with metadata

The page still shows:

- total long-term lookups
- long-term hit rate
- approval queue status

But now it becomes much easier to answer a more precise debugging question:

- did multi-agent memory fail in general?
- or did only user-preference retrieval fail?

## Why This Design Is Better

This round keeps a useful balance:

- user preference memory is no longer missing from multi-agent mode;
- preference context is no longer hidden inside generic long-term memory;
- but preference injection still depends on relevance, so we avoid unnecessary context pollution.

In other words, we now have:

- structured write path
- structured retrieval path
- structured telemetry
- dashboard visibility

without changing the single-agent memory design.

## Validation Notes

Static validation completed:

- `dm_agent/multi_agent/memory.py`
- `dm_agent/memory/long_term_memory.py`
- `dashboard.py`

were parsed successfully with Python AST after modification.

Runtime validation was only partial in this round because:

- the current shell session did not have `OPENAI_API_KEY`, while the default embedding path still expects OpenAI configuration during some memory initialization paths;
- plain system Python in the shell did not match the full CodeAgent conda environment.

So this round confirms the code structure and logic changes, while full runtime verification should continue in the regular `CodeAgent` conda environment with the normal `.env` setup loaded.

## Relationship to Single-Agent Mode

Single-agent mode is unchanged.

This round only improves the multi-agent path:

- centralized preference capture by the orchestrator
- relevant preference injection for sub-agents
- long-term hit recovery
- dashboard-level observability

The original single-agent memory mechanism remains intact for users who choose that mode.

## Next Steps

The next useful improvements are:

1. Add a memory health check panel to the dashboard that compares:
   - vector-store chunk count
   - structured metadata entry count
   - recovered entry count
2. Show a preview of the exact user preference items injected into each task context.
3. Add a lightweight runtime self-check that warns when long-term memory metadata is clearly stale.
4. Continue evaluating whether preference relevance should be profile-aware for domains such as wireless, medical, legal, and finance.
